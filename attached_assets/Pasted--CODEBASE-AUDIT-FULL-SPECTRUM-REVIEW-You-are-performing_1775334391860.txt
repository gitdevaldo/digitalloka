**CODEBASE AUDIT — FULL SPECTRUM REVIEW**

You are performing a comprehensive audit of this codebase. Your job is to find everything wrong, risky, suboptimal, or improvable — then fix what you can directly, and document the rest.

**SCOPE — cover all of the following without exception:**

- **Frontend** — UI consistency, component structure, accessibility, responsiveness, dead code, unused assets, hardcoded values that should be variables
- **UX** — user flows, error states, loading states, empty states, edge case handling, confusing interactions
- **Backend / API** — endpoint logic, error handling, input validation, rate limiting, authentication gaps, missing guards, N+1 queries, inefficient data fetching
- **Database (Supabase)** — connect directly to the project and inspect the live schema. Check: missing indexes, RLS policies (enabled but misconfigured, or missing entirely), foreign key constraints, nullable columns that shouldn't be, data type mismatches, unused tables/columns, query performance issues
- **Security** — exposed secrets, insufficient auth checks, missing CORS config, SQL injection surface, insecure direct object references
- **Performance** — bundle size, render bottlenecks, unoptimized queries, missing caching, unnecessary re-renders
- **Code quality** — duplicated logic, inconsistent naming, missing types, poor separation of concerns, files/functions that are doing too much
- **Future risks** — architectural decisions that will cause pain at scale, missing error boundaries, no retry logic on critical operations, vendor lock-in without abstraction

**DATABASE ACTIONS — execute directly, don't just write SQL:**

If you find database issues (missing index, bad RLS policy, schema problem, etc.), run the fix via Supabase CLI or MFA-authenticated SQL execution. Do not just output the SQL and stop. Confirm execution and capture the result.

**OUTPUT FORMAT per issue:**

- **Area** — which layer (frontend / backend / database / security / etc.)
- **Severity** — Critical / High / Medium / Low
- **Finding** — what the problem is, exactly where it lives (file path, function name, table name)
- **Impact** — what breaks or degrades if left unfixed
- **Action taken** — what you already fixed (with diff or confirmation), or if it requires human decision, why

Prioritize by severity. Fix everything you can autonomously. Flag only what genuinely requires a human call (business logic decisions, destructive schema changes without data migration).
